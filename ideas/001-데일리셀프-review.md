## 4. 기술 검증

### 4-1. 기술 스택 정합성: **통과** (4/5)

| 검증 항목 | 결과 | 소견 |
|----------|------|------|
| RN + TS + Supabase 구현 가능 | 가능 | 모든 MVP 기능이 해당 스택으로 구현 가능. 외부 AI API 의존 없음 |
| 각 기술 선택 적합성 | 양호 | Zustand(경량 상태관리), Expo Push(무료), RevenueCat(1인 최적) 등 적절 |

**발견된 문제점:**
1. **UI 라이브러리 이중 명시**: React Native Paper와 NativeBase가 병렬 기술됨. 하나로 확정 필요 (React Native Paper 권고)
2. **Expo 워크플로우 미명시**: Managed vs Bare 워크플로우 결정이 카카오 SDK 연동 방식에 영향
3. **React Navigation v6 vs v7**: 신규 프로젝트에서 v7 채택 검토 필요

### 4-2. 아키텍처 정합성: **통과** (5/5)

| 검증 항목 | 결과 | 소견 |
|----------|------|------|
| 1인 개발 규모 적합 | 매우 적합 | 서버 관리 부담 제로(Supabase BaaS), 핵심 로직 클라이언트 배치, 외부 의존 최소 |
| 과도한 복잡성 | 없음 | 레이어 분리(Presentation-Logic-Data-Infra)가 1인 개발에 적합한 수준 |
| 로컬 퍼스트 원칙 | 올바름 | 적응형 엔진이 클라이언트에서 동작 → 서버 비용 절감 + 오프라인 가능 |

### 4-3. MVP 범위 적정성: **조건부 통과** (3/5)

12주 로드맵의 기능 범위는 달성 가능하나, 일정 리스크가 존재한다.

**Phase별 리스크 평가:**

| Phase | 기간 | 일정 리스크 | 핵심 우려 |
|-------|------|-----------|----------|
| Phase 1 (1-4주) | 기반 구축 | 낮음~중 | 카카오 OAuth 행정 절차 미반영 |
| Phase 2 (5-8주) | 핵심 기능 | **높음** | 적응형 엔진 + UI + 푸시 알림이 동시 집중. 4주에 과다 |
| Phase 3 (9-12주) | 인사이트+출시 | **높음** | 리포트 + 결제 + QA + 스토어 심사를 4주에 압축. 첫 심사 리젝 시 버퍼 없음 |

**권고: 로드맵 재조정**
```
Phase 1 (1-4주): 셋업 + Auth + 온보딩 + 루틴 CRUD + 엔진 로직(UI 없이)
Phase 2 (5-9주): 체크인 UI + 엔진 통합 + 루틴 실행 + 회고 + 푸시
Phase 3 (10-11주): 주간 리포트(단순 집계) + RevenueCat
Phase 4 (12주+): QA + 스토어 제출 (1-2주 버퍼 확보)
```

**추가 권고:**
- 주간 리포트의 상관관계 분석(PostgreSQL `corr()`)은 v1.1로 연기. MVP는 단순 집계 리포트만 제공
- 적응형 엔진을 Phase 1에서 순수 TS 모듈로 먼저 개발하고, Phase 2에서 UI 통합

### 4-4. DB 설계: **통과** (4/5)

| 검증 항목 | 결과 | 소견 |
|----------|------|------|
| 테이블 구조 | 적절 | 7개 테이블이 핵심 엔티티를 충실히 반영 |
| 과도한 정규화 | 없음 | MVP 규모에 적합 |

**발견된 문제점:**
1. `weekly_reports` 테이블의 조기 물리화 — MVP에서는 View/RPC로 대체 가능
2. `routines` 테이블에 Soft Delete(`deleted_at`) 미적용 — 과거 `plan_items` 참조 깨질 위험
3. `routine_categories` RLS: `user_id IS NULL` (시스템 카테고리) 접근 정책 미비

### 4-5. 오프라인/동기화 전략: **보완 필요** (3/5)

"로컬 퍼스트" 원칙이 선언되어 있으나, 구체적 구현 전략이 부재하다.

**누락 사항:**
- 로컬 저장소 선택 (MMKV, AsyncStorage, SQLite 등) 미명시
- 오프라인→온라인 동기화 전략 (Queue + Retry, Last Write Wins 등) 미명시
- 오프라인 지원 범위 미정의 (어떤 기능까지 오프라인에서 동작하는지)

**권고: MVP에서는 "오프라인 내성(Offline Tolerance)" 수준으로 범위 한정**
```
오프라인 필수 지원:
  - 오늘의 루틴 플랜 조회 (로컬 캐시)
  - 루틴 완료 체크 (로컬 저장 → 온라인 시 동기화)
  - 적응형 엔진 실행 (순수 클라이언트 로직)

온라인 필수:
  - 로그인/회원가입
  - 주간 리포트 조회
  - 구독 관리
```

### 5-3. 측정 가능성: **양호** (4/5)

| 검증 항목 | 결과 | 소견 |
|----------|------|------|
| KPI 구체성 | 우수 | 6개 지표가 명확히 정의됨 (DAU, 체크인 완료율, D7/D30 리텐션, 달성률, 전환율) |
| 목표치 현실성 | 일부 과대 | D7 40%, D30 20%는 업계 평균(D7: 15-25%, D30: 8-15%) 대비 공격적 |
| 북극성 지표 | 방향 적절, 정의 모호 | "주간 체크인 완료율"의 구체적 산식과 목표치 미명시 |

- 리텐션 목표: D7 30%, D30 15%로 보정 (달성 시 상향)
- 북극성 지표 정의: "주 5일 이상 체크인 사용자 비율" + 목표치 명시
- 구독 KPI 추가: MRR, ARPU, Churn Rate

**권고:** 3일차에 "미니 인사이트" 제공하는 Early Win 전략 추가

## 8. 전환 안내

### 판정: **통과 — 5단계 채택 가능**

PRD의 근본적 결함은 없으며, 사업성·기술·문서 품질 모든 영역에서 기준을 충족한다. 발견된 보완 사항은 문서 수정으로 해결 가능한 수준이다.

**다음 단계 옵션:**

1. **즉시 채택** → `/stage-5 001-데일리셀프 채택` (필수 보완 #1~2는 채택 후에도 수정 가능)
2. **보완 후 채택** → PRD 보완 작업 수행 후 `/stage-5 001-데일리셀프 채택`
3. **PRD 수정 후 재판별** → `/stage-3 001-데일리셀프` (권고 보완 사항 전체 반영 후)
