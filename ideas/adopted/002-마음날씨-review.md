## A. 사업성 검증 — **통과**

### 시장 규모·경쟁분석 일치 여부

PRD의 시장 데이터는 2단계 산출물과 **대체로 일치**한다.

- TAM/SAM/SOM: PRD §2-2의 "MZ세대 감정 관리 중간층 300~500만 명"은 시장분석 §1-2의 "감정 기록 앱 잠재 사용자 약 300~500만 명"과 일치
- 한국 기분 기록 앱 성장 데이터 "2023 50만→2025 200만 (4배 성장, 와이즈앱)"이 양 문서에서 동일하게 인용
- 경쟁사 분석(하루콩 1,000만 DL, 루빗 210만+ DL, 라일리 하루 2024.12 출시 등)이 경쟁분석 문서와 일치

**가격 변경 사항**: 2단계 아이디어 문서에서는 "월 3,500원 / 연 27,900원"이었으나, PRD에서는 "월 3,900원 / 연 29,900원"으로 변경. PRD §10-2에서 RevenueCat 2025 벤치마크 기반으로 가격 상향 근거를 제시하고 있어 합리적 변경으로 판단.

### 차별점 방어 가능성

- "기록→분석→행동 완결 루프"는 경쟁사 대비 유의미한 차별점. 하루콩(기록만), 무디(퀘스트 중심), 루빗(습관 중심)과 포지셔닝 분리
- 한국어 감정 어휘 30~40종은 단독으로는 방어력이 낮으나(모방 용이), 데이터 축적 기반 개인화와 결합하면 중기 해자 형성 가능
- PRD §9-3에서 단기/중기/장기 Moat 전략이 구체적으로 명시됨

### 수익 모델

- 프리미엄 구독 (월 3,900원 / 연 29,900원)은 건강&피트니스 앱 벤치마크 대비 합리적 가격대
- 7일 무료 체험(결제 정보 없이)→유료 전환 전략이 RevenueCat 2025 데이터(무료체험→유료 44.5%)에 근거
- 무료/유료 가치 격차가 명확: 월간 상세 리포트(블러 처리), 무제한 셀프케어 카드

## B. 리스크 검증 — **통과**

### 핵심 리스크 식별

PRD §11에서 10개 리스크가 유형(사업/기술/법률/운영)별로 분류되고, 가능성/영향도/심각도/대응 방안이 모두 기술됨.

- **R1(하루콩 AI 기능 추가)**: Critical — 하루콩의 글로벌 전략(해외 80%)으로 한국어 특화 가능성 낮다는 분석이 경쟁분석과 일치
- **R6(감정 기록 습관화 실패)**: Critical — 10초 기록 + 푸시 리마인더 + 셀프케어 카드 즉시 보상으로 대응. North Star Metric을 "주 3회+ 기록자 수"로 설정하여 모니터링 체계 확보
- **R3(낮은 유료 전환율)**: High — 무료체험 + 블러 전략으로 대응하되, 전환율 미달 시 대안(캐릭터/테마 판매 등)은 언급 없음

### 외부 API·플랫폼 의존성

- PRD §12-2에서 외부 의존성 5개(Supabase, RevenueCat, Expo EAS, 카카오 Developer, Apple/Google)를 식별하고 각각 대안을 명시

## C. 기술 검증 — **통과 (Minor 지적 있음)**

### React Native + TypeScript + Supabase 구현 가능성

모든 P0 기능이 해당 스택으로 구현 가능하다. 외부 AI/ML 의존 없이 PostgreSQL SQL 집계로 패턴 분석을 구현하는 접근은 1인 개발에 적합하다.

### 1인 개발 규모 적절성

- 3개월 12주 주별 마일스톤(§14-1)이 현실적으로 구성됨
- P2 기능(분기별 트렌드, PDF 내보내기)은 MVP에서 명시적으로 제외(§13)
- 기술 스택이 Expo Managed Workflow 기반으로 네이티브 빌드 복잡도 최소화

### DB 스키마·API 정합성

- 9개 테이블 정의, FK 관계, 인덱스, RLS 정책이 완전히 작성됨 (§7-5)
- 6개 핵심 DB Function(RPC)의 **완전한 SQL 본문**이 작성됨 (§7-5-5) — 스텁이 아닌 실행 가능한 SQL
- API 설계가 PostgREST REST / RPC / Edge Function으로 3계층 분리되어 있고, 엔드포인트 목록(§7-6-2)과 TypeScript 인터페이스(§7-6-3, 20개)가 완비
- Edge Function 4개의 비즈니스 로직이 단계별로 상세 서술 (§7-6-4)
- 기능-테이블-API 매핑표(§7-8)에서 모든 P0 기능이 매핑됨

### 오프라인 우선 설계

- MMKV + op-sqlite 역할 분담(§7-3-1, §7-3-2)이 데이터 특성에 맞게 설계됨
- 동기화 큐 로직(§7-3-3)이 상태 머신(pending→synced→failed)으로 정의, 지수 백오프 재시도(5회, 최대 30초 cap) 명시
- 충돌 해결은 Server Wins 정책(§7-3-4) — 1인 사용자 앱에서 합리적

### 상태 관리 구조

- 9개 Zustand Store(§7-7)가 역할별로 정의되고, 각 Store의 주요 상태 필드와 persist 전략이 명시됨

### 장애 시나리오 대응

- 오프라인 기록(§7-3-3)으로 네트워크 장애 대응
- Supabase 장애 시 op-sqlite 큐로 서비스 연속성 확보 (R8 대응)
- 결제 실패 시 에러 메시지 + 재시도 (AC-16-4)
- OAuth 인증 실패 시 에러 메시지 표시 (AC-01-5)

### Minor 지적

| # | 항목 | 심각도 | 유형 | 내용 |
|---|------|--------|------|------|
| C-1 | get_calendar_summary SQL 오류 | Minor | 문서 결함 | §7-5-5 RPC 4의 `get_calendar_summary` 함수에서 서브쿼리 내 `FROM UNNEST(array_agg(el2.emotion_tag_ids))` 구문이 PostgreSQL 문법상 불완전하다. 중첩 FROM 절과 상관 서브쿼리의 결합이 올바르지 않으며, 단순히 `emotion_logs`를 날짜로 필터링한 뒤 `UNNEST(emotion_tag_ids)`로 태그별 빈도를 카운트하는 방식으로 수정이 필요하다. 단, 의도(대표 감정 색상 조회)는 명확하고 실행 시 디버깅으로 수정 가능한 수준 |
| C-2 | notification_settings 최대 3개 제한 | Minor | 문서 결함 | §7-5-1에서 `chk_max_notifications` CHECK 제약이 `CHECK (TRUE)`로 비어 있고 "앱 레벨에서 제한"이라고 주석 처리됨. DB 레벨 제약이 없으므로 앱 로직 우회 시 3개 초과 생성 가능. DB function이나 트리거로 제한하는 것이 안전하나, MVP에서는 앱 레벨 제한으로 충분할 수 있음 |
| C-3 | increment_daily_card_usage 경계 조건 | Minor | 문서 결함 | §7-5-5 RPC 6에서 `card_count > 3` 체크 시 이미 INSERT/UPDATE가 실행된 후여서, 한도 초과 시에도 카운트가 증가된 상태. 주석에 "클라이언트에서 사전 체크"라고 명시되어 있어 의도적 설계이나, 한도 초과 시 UPDATE를 롤백하거나 INSERT 전에 체크하는 것이 더 안전함 |
